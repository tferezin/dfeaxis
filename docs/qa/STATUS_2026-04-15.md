### 3.3. Bugs de produção descobertos pelo Agente 1 e CORRIGIDOS por mim ✅

Durante a escrita dos testes T01-T10, o agente descobriu que o código real divergia da spec em 3 pontos. Eu fixei os 3:

**Bug A — Trial bloqueado retornava 403 (Forbidden), não 402 (Payment Required)**
- **Arquivo**: `backend/middleware/security.py` (3 pontos no `verify_trial_active`)
- **Por quê importa**: 402 é o código HTTP padrão pra "precisa pagar". 403 significa "você não tem permissão" (outra coisa). Clientes que olham o código pra decidir o que fazer ficavam confusos.
- **Fix**: troquei 403→402 nos 3 pontos + atualizei docstrings + atualizei os asserts dos 10 testes do T01-T10.
- **Commit**: `22f0a37`

**Bug B (BLOQUEADOR REAL) — `/api/v1/polling/trigger` só aceitava JWT Bearer, não X-API-Key**
- **Arquivo**: `backend/routers/polling.py` e `backend/middleware/security.py`
- **Por quê importa**: você tinha sido explícito: clientes SAP DRC são **on-demand**, chamam a API com API Key. Se o endpoint só aceita JWT de dashboard, a integração SAP DRC não funciona. Esse bug era bloqueador de produto, não de código.
- **Fix**:
  - `/polling/trigger` agora usa `verify_jwt_or_api_key` em vez de `verify_jwt_with_trial`
  - `verify_jwt_or_api_key` foi corrigido pra rodar `verify_trial_active` também no caminho X-API-Key (antes bypassava o gate de trial)
- **Commit**: `22f0a37`

**Bug C — PFX mal-formado retornava 500 genérico**
- **Arquivo**: `backend/scheduler/polling_job.py::_get_pfx_password`
- **Por quê importa**: se a senha do certificado fica corrompida (hex inválido, tag GCM quebrada), o erro propagava como 500 sem mensagem, cliente não sabia o que fazer.
- **Fix**: try/except que captura qualquer erro de decrypt e retorna 502 com `code=CERT_PASSWORD_DECRYPT_FAILED` e mensagem "Reenvie o certificado pela tela Configurações → Certificados".
- **Commit**: `22f0a37`

### 3.6. Bug de produção descoberto pelo Agente 2 — AINDA NÃO CORRIGIDO 🔴

**Bug D — Renewal mensal não zera `docs_consumidos_mes`**
- **Arquivo**: `backend/services/billing/subscriptions.py::sync_subscription_to_db`
- **Sintoma**: quando o Stripe manda webhook `invoice.paid` no fim do ciclo, o código atualiza `current_period_end` e `docs_included_mes` (do plano), mas NÃO zera `docs_consumidos_mes`. O reset só acontece no dia 1 do mês via `scheduler/monthly_overage_job.process_monthly_overage`.
- **Impacto real**: cliente Starter paga dia 15. No mês 2, o ciclo de cobrança vira dia 15 (pós-renewal), mas o contador dele continua em 2800/3000 até dia 1 do mês seguinte. Se ele capturar mais 200 docs nesse período, bate no cap e fica bloqueado mesmo tendo pago.
- **Correção proposta**: zerar `docs_consumidos_mes=0` no handler `_on_invoice_paid` quando detectar transição de `current_period_end`. 1-2 linhas.
- **Por que não corrigi hoje**: é código de billing, quero tua aprovação antes de mexer. É fix pequeno, mas em área crítica.
